# Contributing

Thanks for your interest in contributing to OpenBSP UI!

## Local Setup

Requires Node (the repo pins the version in `mise.toml`).

1. Clone the repo:

   ```bash
   git clone https://github.com/matiasbattocchia/open-bsp-ui
   cd open-bsp-ui
   ```

2. Install dependencies:

   ```bash
   npm install
   ```

3. Configure the environment (point it at a Supabase backend — see
   [open-bsp-api](https://github.com/matiasbattocchia/open-bsp-api)):

   ```bash
   cp .env.example .env
   ```

   Set `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`. `VITE_META_APP_ID` and
   `VITE_FB_LOGIN_CONFIG_ID` are only needed for the WhatsApp connect flow.

4. Run the dev server:

   ```bash
   npm run dev
   ```

   Open [http://localhost:5173](http://localhost:5173).

## Generated & Synced Files

Do not hand-edit these — your changes will be overwritten:

- **`src/routeTree.gen.ts`** — generated by the TanStack Router Vite plugin on
  `npm run dev` / `npm run build`. Adding a file under `src/routes/` regenerates
  it; commit the result.
- **`src/supabase/db_types.ts`** — generated from the database schema and kept
  in sync with the API repo:
  `npx supabase gen types typescript --project-id <ref> > src/supabase/db_types.ts`.

The hand-maintained types in **`src/supabase/types/*`** mirror
`open-bsp-api/supabase/functions/_shared/types/*` (with a few UI-only
divergences). When the API's shared types change, port the change here.

## Code Checks

CI runs `.github/workflows/check.yml` on every push and pull request. Run the
same checks locally before pushing so they pass:

```bash
# 1. Format the whole repo (CI runs `prettier --check .`)
npx prettier --write .

# 2. Lint
npm run lint

# 3. Type-check
npx tsc -b
```

`prettier` formats in place; CI only verifies (`prettier --check .`), so a
commit with unformatted files — including Markdown such as `README.md` — will
fail the check. Run `npx prettier --check .` to preview what would fail without
changing any files.

### Linting policy

ESLint uses typescript-eslint's type-aware **`recommendedTypeChecked`** tier,
with a few rules relaxed in `eslint.config.js` to match the codebase's idioms.
Type-safety gaps (`any`, floating promises) are reported as **warnings** —
tracked debt that does not fail the build. Keep `npm run lint` at **zero
errors**, and avoid adding new warnings where you reasonably can.

## Submitting Changes

1. Fork the repo and create a branch from `develop`
2. Make your changes
3. Run the [code checks](#code-checks) and ensure they pass
4. Open a pull request with a clear description

PRs are welcome for bug fixes, features, and documentation.
